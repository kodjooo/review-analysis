# AGENTS.md — инструкции для AI-агента (Codex и др.)

Основные правила  описаны в 'docs/arch-rules.md'
Бизнес требования к сервису описаны в 'docs/requirements.md'

При изменении проекта обязательно нужно вносить описание архитектуры в документ 'docs/architecture.md' использовать его в работе. Также нужно актуализировать '.env.example', '.env' файлы и тесты в папке 'tests'

После реализации каждого этапа плана из файла 'docs/plan.md' нужно обязательно отмечать его там (писать 'выполнено'), как выполненный, но не удалять

В .env файл пиши комментарии, какой доступ и где нужно найти.

В GITHUB не нужно отправлять .env файл 

По завершению работ пушь изменения в гитхаб в этот репозиторий https://github.com/kodjooo/review-analysis.git, инициируй его, если это еще не сделано. Информацию по разворачиванию проекта на удаленном сервере нужно писать в readme.md  (та там использовать ссылку git pull  - https://github.com/kodjooo/lead-generation-wine.git).
Для этого репозитория используй `origin` по HTTPS и выполняй push через HTTPS-аутентификацию, а не через SSH-ключ.

Для анализа рабочей базы данных на удаленном сервере используй файл `secure/codex-working-db.env`.
В нем лежат параметры readonly-доступа к рабочей PostgreSQL БД.
Когда пользователь просит посмотреть данные именно в рабочей БД, сначала используй эти доступы.
Если прямое подключение по `WORKING_DB_HOST` недоступно, используй внешний IP сервера или SSH-туннель и те же учетные данные.

Для работы с удаленным сервером используй SSH-доступ пользователя `codex`.
Когда пользователь просит что-то посмотреть или проверить именно на сервере (например, логи, статус docker compose, файлы проекта, конфиги, запущенные сервисы), сначала подключайся по SSH под `codex` и выполняй диагностику там.
Если запрос касается рабочей БД на сервере, используй либо SSH-доступ на сервер, либо доступы из `secure/codex-working-db.env` — в зависимости от того, что удобнее и надежнее для задачи.
Для обычных задач на сервере используй только пользователя `codex`.
Не подключайся по SSH под `root`, если пользователь отдельно и явно не попросил использовать именно `root`.
На удаленно сервере дирректория проекта это `/usr/local/automation/review-analysis/`
