ТЗ на автоматизацию мониторинга отзывов и рейтингов по Яндекс и 2ГИС

1. Общая задача

Необходимо разработать автоматизацию для регулярного мониторинга карточек магазинов / объектов на площадках:
- Яндекс
- 2ГИС

Система должна по списку объектов:
- заходить на карточки отзывов;
- собирать текущие данные;
- сравнивать их с предыдущим сохраненным состоянием;
- формировать отчет;
- отправлять отчет на e-mail.

2. Основная цель

Автоматизация должна помогать быстро видеть по каждому магазину:
- сколько было отзывов на предыдущем замере;
- сколько стало сейчас;
- сколько появилось новых отзывов;
- изменилась ли средняя оценка;
- есть ли новые отзывы с оценкой ниже 5 звезд;
- текст этих новых негативных / условно негативных отзывов.

3. Источники мониторинга

В первой версии обязательно поддержать:
- Яндекс Карты / Яндекс Отзывы
- 2ГИС / отзывы в 2ГИС

Примеры входных ссылок

Яндекс:
https://yandex.com/maps/org/vinoteka_kul_tura_krepkogo/147946479764/reviews/

2ГИС:
https://2gis.ru/krasnodar/firm/70000001104623156/tab/reviews

4. Входные данные

Система должна получать на вход конфигурационный файл со списком магазинов / объектов для мониторинга.

Обязательные поля по каждому объекту:
- тип магазина
- адрес
- ссылка на Яндекс
- ссылка на 2ГИС

Желательно также предусмотреть:
- внутреннее название магазина / точки;
- уникальный ID точки;
- флаг активности (включен / выключен мониторинг).

5. Формат конфигурационного файла

Нужно предусмотреть хранение входных данных в отдельном конфиге, который можно редактировать без изменения кода.

Подходящие варианты:
- CSV
- Excel
- JSON
- YAML

Предпочтительно:
Использовать формат, который легко редактировать вручную. Практически удобнее всего:
- CSV / Excel — для менеджера;
- JSON / YAML — для разработчика.

Пример структуры в CSV

id | type | address | yandex_url | twogis_url | is_active
1 | Винотека | Краснодар, ул. ... | https://yandex... | https://2gis... | 1
2 | Магазин | Краснодар, ул. ... | https://yandex... | https://2gis... | 1

Пример структуры в JSON

[
  {
    "id": "store_001",
    "type": "Винотека",
    "address": "Краснодар, ул. Красная, 10",
    "yandex_url": "https://yandex.com/maps/org/vinoteka_kul_tura_krepkogo/147946479764/reviews/",
    "twogis_url": "https://2gis.ru/krasnodar/firm/70000001104623156/tab/reviews",
    "is_active": true
  }
]

6. Что нужно мониторить по каждой площадке

Для каждой ссылки Яндекс и 2ГИС нужно собирать:
1. общее количество отзывов на текущий момент
2. среднюю оценку на текущий момент
3. последние N отзывов
4. для каждого отзыва:
   - дата;
   - количество звезд;
   - текст;
   - ссылка на карточку;
   - при наличии — имя автора;
   - при наличии — уникальный ID отзыва или иной идентификатор.

7. Принцип работы

7.1. Общая логика

Система не должна анализировать всю историю отзывов за все время.
Она должна работать по модели сравнения текущего замера с предыдущим.

То есть на каждом запуске:
- получить текущее состояние карточки;
- получить последние N отзывов;
- сравнить их с предыдущим сохраненным состоянием;
- определить новые отзывы;
- зафиксировать изменение общего числа отзывов и рейтинга.

7.2. Сравнение по площадкам

Мониторинг должен вестись раздельно по Яндексу и 2ГИС, потому что:
- количество отзывов на площадках разное;
- рейтинги разные;
- тексты отзывов разные;
- новые отзывы могут появиться только на одной из площадок.

Следовательно:
в отчете нужно показывать данные отдельно по:
- Яндекс
- 2ГИС

8. Как должна выглядеть сущность магазина в системе

Для одной точки должен существовать один объект мониторинга, внутри которого две площадки:
- Яндекс
- 2ГИС

То есть логика должна быть такой:

Точка / магазин
- тип магазина
- адрес
- Яндекс: данные
- 2ГИС: данные

9. Ограничение на количество анализируемых отзывов

Необязательно собирать всю ленту отзывов.

Нужно предусмотреть параметр, который определяет, сколько последних отзывов анализируется на каждой площадке.

Например:
- 10
- 20
- 30

Требование:
Количество должно задаваться в настройках.

Например:
review_fetch_limit_yandex = 20
review_fetch_limit_twogis = 20

или единым параметром:
review_fetch_limit = 20

10. Что считать новым отзывом

Новый отзыв — это отзыв, которого не было в предыдущем замере по соответствующей площадке.

Сравнение делать по одному из вариантов:
1. ID отзыва, если доступен;
2. если ID нет — по комбинации:
   - дата;
   - оценка;
   - текст;
   - автор (если есть).

Нужно предусмотреть устойчивый способ дедупликации, чтобы один и тот же отзыв не считался новым повторно.

11. Данные, которые нужно хранить

Для корректной работы системы необходимо хранить историю предыдущих замеров.

По каждой точке и каждой площадке сохранять:
- дата и время замера;
- общее количество отзывов;
- средняя оценка;
- список последних N отзывов;
- список идентификаторов / сигнатур отзывов;
- результат сравнения с предыдущим замером.

12. Формат хранения

Допустимые варианты:
- SQLite
- PostgreSQL
- JSON-файлы
- Google Sheets
- Excel / CSV

Предпочтительно для первой версии:
Использовать SQLite или аналогичное простое локальное хранилище, чтобы:
- не зависеть от внешних сервисов;
- хранить историю замеров;
- быстро сравнивать старое и новое состояние.

13. Логика формирования отчета

Отчет должен отправляться на e-mail и включать два уровня.

13.1. Сводка по каждой точке

По каждой точке показать:
- тип магазина;
- адрес;
- Яндекс: было / стало / новых / рейтинг был / рейтинг стал;
- 2ГИС: было / стало / новых / рейтинг был / рейтинг стал.

Пример логики:

Винотека, Краснодар, ул. ...

Яндекс
- было отзывов: 98
- стало отзывов: 100
- новых: 2
- рейтинг был: 4.7
- рейтинг стал: 4.6

2ГИС
- было отзывов: 41
- стало отзывов: 42
- новых: 1
- рейтинг был: 4.5
- рейтинг стал: 4.5

13.2. Блок с новыми отзывами ниже 5 звезд

После сводки выводить только новые отзывы, у которых оценка:
- 4
- 3
- 2
- 1

Для каждого такого отзыва указывать:
- точку / магазин;
- площадку;
- дату;
- количество звезд;
- текст;
- ссылку на карточку.

14. Структура итогового письма

Тема письма:
Мониторинг отзывов за неделю — [дата]

Тело письма:
1. Общая сводка по всем точкам
2. По каждой точке:
   - тип магазина
   - адрес
   - статистика по Яндекс
   - статистика по 2ГИС
3. Отдельный раздел:
   - новые отзывы ниже 5 звезд
4. Ссылки на карточки для ручной проверки

15. Предпочтительный формат таблицы в письме

Сводная таблица

Тип магазина | Адрес | Площадка | Было отзывов | Стало отзывов | Новых | Рейтинг был | Рейтинг стал | Ссылка

Таблица новых отзывов ниже 5

Тип магазина | Адрес | Площадка | Дата | Звезды | Текст | Ссылка

16. Периодичность запуска

На первом этапе нужен регулярный автоматический запуск, базово — раз в неделю.

При этом нужно предусмотреть:
- возможность ручного запуска;
- возможность позже перевести на ежедневный мониторинг без переписывания логики.

17. Обработка нестандартных ситуаций

Если по площадке данные не удалось собрать, это не должно ломать весь отчет.

В таком случае:
- по конкретной площадке ставится статус ошибки;
- в письме отображается пометка:
  - не удалось получить данные;
  - источник временно недоступен;
  - требуется повторная проверка.

Пример:
- Яндекс — успешно
- 2ГИС — ошибка получения данных

18. Что не требуется в первой версии

Не нужно делать в первой версии, если это усложняет реализацию:
- анализ всех исторических отзывов за все время;
- поиск удаленных старых отзывов на уровне конкретного текста;
- сложную аналитику тональности;
- автоответы на отзывы;
- отдельный интерфейс управления;
- объединение отзывов Яндекса и 2ГИС в одну общую оценку.

19. Обязательные настройки

В конфигурации системы должны настраиваться:
- список точек;
- тип магазина;
- адрес;
- ссылка на Яндекс;
- ссылка на 2ГИС;
- флаг активности;
- e-mail получателя;
- число последних отзывов для анализа;
- порог звезд для вывода текста в отчет;
- расписание запуска.

20. Минимальный обязательный результат

Система считается готовой, если она:
1. читает конфигурационный файл со списком точек;
2. умеет обрабатывать отдельно Яндекс и 2ГИС;
3. собирает по каждой площадке:
   - число отзывов,
   - рейтинг,
   - последние N отзывов;
4. сравнивает текущее состояние с предыдущим;
5. определяет новые отзывы;
6. выделяет новые отзывы ниже 5 звезд;
7. отправляет письмо с понятной сводкой.

21. Acceptance criteria

Разработка считается выполненной, если на тестовом наборе ссылок система:
- принимает конфиг с полями:
  - тип магазина,
  - адрес,
  - ссылка на Яндекс,
  - ссылка на 2ГИС;
- корректно проходит по всем активным точкам;
- получает данные минимум по одной и той же тестовой точке с обеих площадок;
- считает “было / стало / новых” по каждой площадке;
- показывает рейтинг “был / стал”;
- выводит тексты новых отзывов с оценкой ниже 5;
- отправляет итоговый отчет на e-mail;
- не падает целиком, если одна из ссылок недоступна.

22. Краткая постановка для разработчика

Нужно сделать автоматизацию мониторинга отзывов для точек, где на вход подается конфигурационный файл со списком объектов: тип магазина, адрес, ссылка на Яндекс, ссылка на 2ГИС. Система должна регулярно заходить в карточки отзывов на обеих площадках, собирать текущее число отзывов, среднюю оценку и последние N отзывов, сравнивать это с предыдущим замером, считать новые отзывы и отправлять на почту отчет. В отчете по каждой точке нужно отдельно показать Яндекс и 2ГИС: было / стало / новых / рейтинг был / рейтинг стал. Ниже отдельным блоком вывести тексты только новых отзывов с оценкой ниже 5 звезд.
