План реализации сервиса мониторинга отзывов

Общие принципы:
- Вся разработка, запуск, тестирование и отладка выполняются только через Docker Desktop.
- Все компоненты проекта должны запускаться через основной `docker-compose.yml`.
- Первая версия делается максимально простой: без отдельной панели управления и без анализа всей исторической ленты отзывов.
- Основное хранилище первой версии: SQLite в Docker volume.

Этап 1. Подготовка каркаса проекта и Docker-инфраструктуры — выполнено
- Выполнено, когда в проекте есть базовая Docker-схема для локальной разработки и запуска только через Docker Desktop.
- Создать и описать основной `docker-compose.yml` для приложения.
- Определить состав контейнеров первой версии:
  - контейнер приложения;
  - при необходимости отдельный контейнер для одноразовых задач или тестов;
  - постоянный volume для SQLite и служебных файлов.
- Привести `Dockerfile` к рабочему сценарию запуска сервиса и фоновых/ручных команд.
- Зафиксировать единый рабочий каталог внутри контейнера и схему подключения конфигов/данных через volumes.
- Подготовить команды для:
  - сборки образа;
  - ручного запуска мониторинга;
  - запуска тестов;
  - просмотра логов.
- Обновить `README.md` только в части реального запуска через Docker Desktop.
- После завершения этапа отметить этот этап в документе словом `выполнено`.

Этап 2. Архитектурная схема и структура модулей — выполнено
- Выполнено, когда в `docs/architecture.md` описана целевая структура сервиса и потоки данных первой версии.
- Описать основные доменные сущности:
  - точка мониторинга;
  - площадка;
  - отзыв;
  - снимок состояния площадки;
  - результат сравнения;
  - отчет.
- Разделить приложение на понятные модули:
  - загрузка конфигурации;
  - orchestration сценария запуска;
  - адаптеры площадок Яндекс и 2ГИС;
  - слой хранения SQLite;
  - сервис сравнения снимков;
  - сервис формирования отчета;
  - сервис отправки e-mail;
  - логирование и обработка ошибок.
- Зафиксировать, как данные проходят путь от конфигурации до письма.
- Отдельно описать, как сервис работает при частичном отказе одной площадки.
- После завершения этапа отметить этот этап в документе словом `выполнено`.

Этап 3. Конфигурация проекта и переменные окружения — выполнено
- Выполнено, когда сервис можно настроить без изменения кода через конфиг точек и `.env`.
- Выбрать формат входного списка точек для первой версии.
- Рекомендуемый вариант первой версии:
  - JSON или YAML для точек мониторинга;
  - `.env` для инфраструктурных настроек.
- Реализовать схему конфигурации с обязательными полями:
  - `id`;
  - `type`;
  - `address`;
  - `yandex_url`;
  - `twogis_url`;
  - `is_active`.
- Добавить общие настройки:
  - `review_fetch_limit`;
  - при необходимости отдельные лимиты по площадкам;
  - порог звезд для включения текста в отчет;
  - e-mail получателей;
  - SMTP-параметры;
  - таймауты;
  - настройки SQLite;
  - расписание запуска.
- Обновить `.env.example` и `.env` с комментариями, где брать каждое значение и какой доступ нужен.
- Исключить `.env` из публикации в GitHub.
- Добавить валидацию конфигурации при старте контейнера.
- После завершения этапа отметить этот этап в документе словом `выполнено`.

Этап 4. Модель данных и SQLite-хранилище — выполнено
- Выполнено, когда сервис умеет сохранять и читать предыдущие замеры для сравнения.
- Спроектировать SQLite-схему первой версии.
- Предусмотреть таблицы или эквивалентные структуры для:
  - точек мониторинга;
  - запусков мониторинга;
  - снимков по площадкам;
  - отзывов из последнего снимка;
  - сигнатур отзывов;
  - результатов сравнения;
  - статусов ошибок по площадкам.
- Определить правила идентификации нового отзыва:
  - по `external_review_id`, если он доступен;
  - иначе по стабильной сигнатуре из даты, рейтинга, текста и автора.
- Реализовать слой репозитория для:
  - записи нового снимка;
  - получения последнего успешного снимка;
  - поиска уже известных отзывов;
  - сохранения итогов запуска.
- Настроить хранение файла SQLite в Docker volume, чтобы данные не терялись между перезапусками контейнеров.
- После завершения этапа отметить этот этап в документе словом `выполнено`.

Этап 5. Интеграция с источниками отзывов — выполнено
- Выполнено, когда сервис получает отзывы по Яндексу и 2ГИС отдельно в едином внутреннем формате.
- Определить технический способ получения данных по площадкам.
- Для каждой площадки реализовать отдельный адаптер, который умеет:
  - открыть карточку отзывов;
  - получить общее количество отзывов;
  - получить средний рейтинг;
  - получить последние `N` отзывов;
  - нормализовать отзыв в единый формат.
- Учесть, что у площадок могут отличаться:
  - структура страницы;
  - наличие ID отзыва;
  - формат даты;
  - глубина загрузки отзывов;
  - антибот-ограничения и таймауты.
- Добавить повторные попытки, ограничение ожидания и понятные сообщения об ошибках по каждой площадке.
- Сделать так, чтобы ошибка по одной площадке не останавливала обработку остальных точек и площадок.
- После завершения этапа отметить этот этап в документе словом `выполнено`.

Этап 6. Логика сравнения снимков и выделение новых отзывов — выполнено
- Выполнено, когда сервис считает `было / стало / новых` и определяет новые отзывы ниже заданного порога звезд.
- Реализовать сравнение текущего снимка с предыдущим отдельно по каждой площадке.
- Рассчитывать:
  - количество отзывов было/стало;
  - рейтинг был/стал;
  - число новых отзывов;
  - список новых отзывов;
  - список новых отзывов с рейтингом ниже порога.
- Зафиксировать поведение для первого запуска, когда предыдущего снимка еще нет.
- Учесть частичные сценарии:
  - прошлый снимок есть только по одной площадке;
  - текущий сбор удался только по одной площадке;
  - карточка временно недоступна;
  - отзыв не имеет стабильного внешнего ID.
- После завершения этапа отметить этот этап в документе словом `выполнено`.

Этап 7. Формирование отчета и отправка e-mail — выполнено
- Выполнено, когда сервис формирует итоговое письмо по требованиям и отправляет его получателям.
- Реализовать сборщик отчета с двумя уровнями:
  - общая сводка по точкам и площадкам;
  - отдельный блок новых отзывов с оценкой ниже 5 или ниже настраиваемого порога.
- Подготовить удобный HTML-формат письма с таблицами:
  - тип магазина;
  - адрес;
  - площадка;
  - было;
  - стало;
  - новых;
  - рейтинг был;
  - рейтинг стал;
  - ссылка.
- Для блока негативных новых отзывов выводить:
  - точку;
  - площадку;
  - дату;
  - рейтинг;
  - текст;
  - ссылку.
- Добавить в письмо явные пометки по площадкам, где произошла ошибка сбора данных.
- Реализовать SMTP-отправку и логирование результата отправки.
- После завершения этапа отметить этот этап в документе словом `выполнено`.

- Реализовать планировщик для регулярного запуска.
- Для первой версии выбрать простой и предсказуемый вариант:

  - или отдельный контейнер с cron, который вызывает команду приложения.
- Настроить расписание по умолчанию на еженедельный запуск.
- Предусмотреть возможность переключить частоту на ежедневную через конфигурацию без переписывания кода.
- Описать сценарий запуска только через Docker Desktop.
