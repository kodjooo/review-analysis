1. Назначение системы

Сервис предназначен для регулярного мониторинга отзывов и рейтингов по точкам на площадках Яндекс и 2ГИС, сравнения текущего замера с предыдущим состоянием и отправки e-mail-отчета.

2. Базовые принципы

- Разработка, запуск и тестирование выполняются только через Docker Desktop.
- Основной способ локального запуска: `docker compose`.
- Первая версия не содержит отдельной панели управления.
- Основное хранилище состояния: SQLite в Docker volume.

3. Контейнерная схема

- `app`:
  - одноразовый ручной запуск мониторинга;
  - использует `.env`;
  - читает конфигурацию точек из `config/points.json`;
  - пишет SQLite в `data/`.
- `scheduler`:
  - запускает тот же код приложения в режиме планировщика;
  - использует ту же конфигурацию и тот же volume данных.
- `tests`:
  - запускает автотесты внутри Docker Desktop.

4. Точка входа приложения

- CLI-точка входа: `python -m app.main`
- Поддерживаемые команды:
  - `run` — выполнить один цикл мониторинга;
  - `schedule` — запустить бесконечный цикл планировщика;
  - `test-email` — отправить тестовое письмо;
  - `init-db` — инициализировать SQLite.

5. Архитектурные слои первой версии

- `app.cli`:
  - разбор CLI-аргументов;
  - запуск нужного режима приложения.
- `app.bootstrap`:
  - сборка зависимостей приложения;
  - создание настроек, логгера, БД и сервисов.
- `app.core`:
  - доменные модели;
  - загрузка конфигурации;
  - общие утилиты;
  - логирование.
- `app.db`:
  - подключение к SQLite;
  - схема таблиц;
  - репозитории.
- `app.adapters`:
  - изолированные адаптеры площадок Яндекс и 2ГИС.
- `app.services`:
  - orchestration одного запуска;
  - сравнение снимков;
  - построение отчета;
  - отправка e-mail;
  - планировщик.

6. Доменные сущности

- `MonitoringPoint`:
  - описывает точку мониторинга;
  - содержит тип, адрес, ссылки на площадки и флаг активности.
- `Review`:
  - единый внутренний формат отзыва независимо от площадки;
  - хранит дату, рейтинг, текст, автора, ссылку, внешний ID и сигнатуру.
- `PlatformSnapshot`:
  - результат одного сбора по точке и площадке;
  - хранит количество отзывов, рейтинг, список последних отзывов и статус сбора.
- `PlatformDelta`:
  - результат сравнения текущего и предыдущего снимка;
  - хранит `было / стало / новых` и негативные новые отзывы.
- `PointReport`:
  - объединяет результаты по Яндекс и 2ГИС внутри одной точки.
- `MonitoringRunResult`:
  - итог одного полного запуска мониторинга.
- `EmailMessage`:
  - собранное письмо в HTML и plain text форматах.

7. Конфигурация

Система использует два источника конфигурации:
- `.env`:
  - пути;
  - SMTP;
  - расписание;
  - лимиты;
  - логирование.
- `config/points.json`:
  - список точек;
  - ссылки на Яндекс и 2ГИС;
  - флаг активности.

На уровне приложения конфигурация собирается в объект `Settings`.
При старте выполняется валидация:
- лимитов и таймаутов;
- расписания;
- SMTP-настроек;
- уникальности `id` точек;
- структуры файла точек.

8. Хранение данных

На текущем этапе зафиксирована рабочая схема хранения:
- таблица `monitoring_runs`:
  - информация о запуске мониторинга;
- таблица `platform_snapshots`:
  - снимок состояния точки по конкретной площадке;
- таблица `reviews`:
  - отзывы, попавшие в конкретный снимок.

SQLite-файл хранится внутри `data/`, а сама папка подключена к постоянному Docker volume.

9. Работа репозитория

Репозиторий `MonitoringRepository` отвечает за:
- создание записи нового запуска;
- завершение запуска со статусом;
- получение предыдущего успешного снимка по точке и площадке;
- сохранение нового снимка и связанных отзывов.

Сравнение новых отзывов строится на сигнатурах:
- если у площадки есть внешний ID, он участвует в сигнатуре;
- если ID нет, используется стабильный хеш из даты, рейтинга, текста и автора.

10. Поток выполнения

Базовый поток одного запуска:
1. CLI читает команду и загружает настройки.
2. Приложение инициализирует SQLite.
3. Сервис мониторинга проходит по активным точкам.
4. Для каждой точки отдельно обрабатываются Яндекс и 2ГИС.
5. Адаптер площадки получает HTML страницы и приводит отзывы к единому виду.
6. Сервис сравнения вычисляет разницу с предыдущим успешным снимком.
7. Репозиторий сохраняет новый снимок и отзывы.
8. Построитель отчета формирует HTML и текстовую версию письма.
9. E-mail сервис отправляет итоговый отчет, если SMTP настроен.

11. Обработка частичных отказов

Система проектируется так, чтобы ошибка одной площадки не ломала весь запуск:
- адаптер возвращает ошибочный снимок со статусом `error`;
- сравнение для такой площадки формирует ошибочный `PlatformDelta`;
- остальные площадки и точки продолжают обрабатываться;
- письмо содержит явную пометку об ошибке сбора данных;
- запуск в целом может завершиться статусом `completed_with_errors`.

12. Интеграция с площадками

Для каждой площадки используется отдельный адаптер:
- `YandexAdapter`;
- `TwoGisAdapter`.

Общая логика вынесена в `BaseReviewAdapter`:
- загрузка HTML через Playwright Chromium;
- поддержка локальных `file://` фикстур для тестов;
- повторная попытка открытия страницы;
- нормализация отзыва в общий формат;
- генерация сигнатуры отзыва.

Парсинг площадок устроен по принципу graceful degradation:
- сначала выполняется попытка извлечь агрегированный рейтинг и отзывы из JSON/JSON-LD;
- если этого недостаточно, используются HTML-селекторы и запасные эвристики.

13. Сравнение снимков

`SnapshotComparisonService` сравнивает текущий снимок с предыдущим успешным снимком по точке и площадке:
- считает `было / стало`;
- сравнивает рейтинг `был / стал`;
- определяет новые отзывы по сигнатурам;
- выделяет новые отзывы со звездами ниже или равными порогу из настроек.

Поведение первого запуска:
- если предыдущего снимка нет, все найденные отзывы текущего снимка считаются новыми.

14. Формирование письма

`ReportBuilder` формирует два представления:
- HTML-письмо для реальной отправки;
- plain text версию для совместимости и диагностики.

Структура письма:
- общая таблица по точкам и площадкам;
- блок новых отзывов ниже порога;
- ссылки на карточки площадок;
- статус ошибки по площадке, если сбор не удался.

15. Способы запуска

Поддерживаются следующие режимы:
- `run`:
  - один цикл мониторинга;
  - используется для ручного запуска через Docker Desktop.
- `schedule`:
  - встроенный планировщик;
  - поддерживает `weekly` и `daily`;
  - защищен от повторного запуска в одну и ту же минуту.
- `test-email`:
  - отправка тестового письма без реального цикла мониторинга.
- `init-db`:
  - инициализация SQLite-схемы.

16. Наблюдаемость и ошибки

Логирование организовано через стандартный `logging`:
- настраиваемый уровень через `.env`;
- единый формат строки лога;
- сообщения о старте, обработке точек, ошибках площадок, SMTP и завершении запуска.

Ошибки делятся на два класса:
- локальные ошибки площадки:
  - превращаются в `error`-снимок и попадают в отчет;
  - не останавливают обработку остальных площадок и точек;
- критические ошибки запуска:
  - помечают запуск как `failed`;
  - прерывают текущий цикл мониторинга.

17. Тестовая стратегия

Автотесты запускаются только через Docker Desktop.

Покрытые сценарии:
- загрузка и валидация конфигурации;
- парсинг фикстур Яндекс и 2ГИС;
- сравнение снимков и выделение новых отзывов;
- формирование письма;
- сохранение и чтение SQLite-снимков.

Для тестов используются локальные HTML-фикстуры, чтобы не зависеть от внешних сайтов.
